# Task 9.1 – ServiceNow Incident & Request Management

---

# 1. Create an Incident in ServiceNow

Navigate to:

```text id="sn1"
Incident → Create New
```

---

## Mandatory Fields

Fill in the required fields:

* Caller
* Category (Linux / Windows)
* Short Description
* Priority

---

## Assignment Configuration

Set the following values:

| Field            | Value                     |
| ---------------- | ------------------------- |
| Assignment Group | Linux L2 Support          |
| Impact           | 1-High / 2-Medium / 3-Low |
| Urgency          | 1-High / 2-Medium / 3-Low |

---

## Attach Evidence

Attach relevant supporting information:

* Logs
* Screenshots
* Monitoring alerts
* System outputs

---

## SLA Configuration

Set or verify SLA tracking:

```text id="sn2"
SLA → Applicable SLA → Start (Auto or Manual)
```

Ensure SLA timers are active and correctly associated with the incident.

---

# 2. Work Notes vs Customer Notes

| Type                    | Visibility              | Purpose                                             |
| ----------------------- | ----------------------- | --------------------------------------------------- |
| **Work Notes**          | Internal (IT Team only) | Technical updates, troubleshooting steps, decisions |
| **Additional Comments** | Visible to customer     | End-user communication and status updates           |

---

## Update Guidelines

* Update **Work Notes every 30 minutes** for active P1/P2 incidents.
* Include the following in Work Notes:

  * Commands executed
  * Outputs observed
  * Troubleshooting decisions
  * Progress updates

# Task 9.2 – ARCON PAM Operations

---

# 3. Privileged Access via ARCON

Access privileged systems securely using ARCON PAM.

---

## Login to ARCON Portal

```text id="ar1"
https://arcon.company.internal
```

---

## Request Privileged Access

Navigate to:

```text id="ar2"
My Requests → Request Privileged Account
```

Wait for approval before initiating sessions.

---

## Launch Secure Session

Once approved:

* Launch SSH session via ARCON PAM
* All activity is recorded (keystrokes + screen capture)

---

## Operational Guidelines

* 🛡️ All sessions are fully monitored and recorded.
* ⏱️ Sessions may timeout automatically.
* 🔁 Reconnect if disconnected and document the interruption in incident Work Notes.
* 🔚 Always release privileged access after task completion.
* 🚪 Close ARCON session properly to ensure audit compliance.

# 5. Access Management – User Provisioning & Audit

Manage Linux user accounts and enforce access control policies.

---

## Create User with Sudo Access

### Create User Account

```bash id="usr1"
useradd -m -s /bin/bash -c "John Doe" jdoe
```

---

### Set Password

```bash id="usr2"
passwd jdoe
```

---

### Grant Sudo Access (RHEL)

```bash id="usr3"
usermod -aG wheel jdoe
```

---

### Grant Sudo Access (Ubuntu)

```bash id="usr4"
usermod -aG sudo jdoe
```

---

## Lock / Unlock User Accounts

```bash id="usr5"
usermod -L jdoe   # Lock account
```

```bash id="usr6"
usermod -U jdoe   # Unlock account
```

---

## Audit Sudo Activity

### RHEL

```bash id="aud1"
grep sudo /var/log/secure | tail -30
```

---

### Ubuntu

```bash id="aud2"
grep sudo /var/log/auth.log | tail -30
```

# ✅ Operational Best Practices

Follow these security and operational standards during all administrative activities.

---

## 🔐 Authorization Requirement

Every privileged action must be backed by an approved **ServiceNow ticket**.

> No exceptions.

---

## 🧾 ARCON Session Compliance

* All ARCON sessions are fully recorded (screen + keystrokes)
* Never enter passwords for other systems inside ARCON terminals
* Maintain professional and audit-compliant behavior at all times

---

## 🛡️ Endpoint Security Monitoring

* Review CrowdStrike console **weekly**
* Monitor detection events and anomalies
* Investigate alerts on managed servers proactively

---

## 👤 User Lifecycle Management

* Remove user accounts within **24 hours of off-boarding**
* Failure to do so may result in **security compliance violations**

---

## 📊 CMDB Compliance

* Maintain accurate records of service accounts in **ServiceNow CMDB**
* Link accounts to their respective applications and owners

---

## 🚫 No Shared Accounts

* Every engineer must use a **unique named account**
* Shared privileged accounts are strictly prohibited
