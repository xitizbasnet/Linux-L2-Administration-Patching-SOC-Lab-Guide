## Overview

This lab provides standard operating procedures for applying operating system patches, implementing Linux server hardening, and performing security compliance checks in production environments. It follows industry best practices, including **CIS Benchmarks**, structured **Change Management**, and **Vulnerability Management** processes to ensure system security and stability.

---

# 🎯 Objective

Perform Linux server maintenance by:

* Applying operating system security patches
* Implementing Linux server hardening
* Verifying post-patch system health
* Performing security compliance checks
* Maintaining client-approved security baselines

---

# 📌 Scope

This lab applies to the following Linux platforms and security standards.

# Task 4.1 – Pre-Patching Preparation

Before applying updates, ensure that all change management and backup procedures have been completed.

---

# 1. Create a Snapshot or Backup

Before performing any patching activity:

* Raise a **Change Request** in **ServiceNow**.
* Include:

  * Patch details
  * Maintenance window
  * Rollback plan
  * Business impact
  * Validation steps

---

## Create a Virtual Machine Snapshot

### VMware

Use the **vSphere Client** or VMware CLI.

```bash id="m51x2v"
# VMware
vmware-cmd snapshot
```

---

### Amazon EC2 (AWS)

```bash id="gupwpf"
aws ec2 create-snapshot --volume-id vol-XXXXXX --description "pre-patch"
```

---

### Verification

Before proceeding:

* Verify the snapshot completed successfully.
* Confirm backup integrity.
* Ensure rollback procedures have been documented.

> **⚠️ Important**
>
> Never begin patching until a verified backup or VM snapshot is available.

---

# 2. Check Available Updates

Review available package updates before applying patches.

## RHEL / CentOS

### Check Available Updates

```bash id="yb7bdc"
yum check-update
```

```bash id="9b4zcs"
dnf check-update
```

---

## Ubuntu

Refresh package repositories and list available upgrades.

```bash id="vfw9g6"
apt update && apt list --upgradable
```

---

## Check Security Updates Only

### RHEL / CentOS

```bash id="rf5phg"
yum updateinfo list security
```

---

### Ubuntu

```bash id="tmtvc8"
apt list --upgradable | grep -i security
```

Only security-related packages will be displayed.

# Task 4.2 – Applying Operating System Patches

Apply updates only during an approved maintenance window.

---

# 3. Apply Security Updates (RHEL / CentOS)

## Install Security Updates Only

```bash id="vnwh8o"
yum update --security -y
```

---

## Install All Approved Updates

```bash id="m2jmmz"
yum update -y
```

```bash id="3z5glo"
dnf upgrade -y
```

---

## Check if a Reboot Is Required

```bash id="x1v0z7"
needs-restarting -r
```

Review the output before scheduling a reboot.

---

# 4. Apply Updates (Ubuntu)

Refresh package repositories.

```bash id="rhtxgq"
apt update
```

Upgrade installed packages.

```bash id="svvcxq"
apt upgrade -y
```

Apply distribution-level upgrades.

```bash id="u6jjlwm"
apt dist-upgrade -y
```

Preview unattended upgrades without applying changes.

```bash id="zvvtk6"
unattended-upgrades --dry-run
```

---

## Verify Reboot Requirement

```bash id="h1uf78"
ls /var/run/reboot-required 2>/dev/null && echo "REBOOT NEEDED"
```

If the file exists, schedule a reboot according to the approved maintenance window.

# Task 4.3 – Linux Server Hardening (CIS Benchmark)

Apply security hardening in accordance with organizational standards and CIS Benchmark recommendations.

---

# 6. Disable Unused Services

Disable services that are not required by the operating environment.

```bash id="u8b79e"
systemctl disable avahi-daemon cups rpcbind --now
```

---

## Audit Enabled Services

```bash id="vf4lwv"
systemctl list-unit-files --type=service | grep enabled
```

Review all enabled services and disable unnecessary components where appropriate.

---

# 7. Harden SSH Configuration

Modify the SSH daemon configuration to improve server security.

## Disable Root Login

```bash id="m5esiu"
sed -i 's/^#PermitRootLogin.*/PermitRootLogin no/' /etc/ssh/sshd_config
```

---

## Disable Password Authentication

```bash id="h8r50h"
sed -i 's/^#PasswordAuthentication.*/PasswordAuthentication no/' /etc/ssh/sshd_config
```

---

## Limit Authentication Attempts

```bash id="v6mgzx"
echo 'MaxAuthTries 3' >> /etc/ssh/sshd_config
```

---

## Configure Client Timeout

```bash id="jyk6zf"
echo 'ClientAliveInterval 300' >> /etc/ssh/sshd_config
```

---

## Restart SSH Service

```bash id="5g6wsm"
systemctl restart sshd
```

> **Best Practice**
>
> Validate the SSH configuration before restarting the service to avoid accidental lockout.

---

# 8. Configure Firewall & SELinux

Verify firewall configuration and ensure SELinux is operating in **Enforcing** mode.

---

## Check Firewall Status

```bash id="q3dixg"
firewall-cmd --state
```

---

## Display Active Firewall Rules

```bash id="7l53gh"
firewall-cmd --list-all
```

---

## Allow SSH Service

```bash id="k9jry9"
firewall-cmd --permanent --add-service=ssh --zone=public
```

---

## Reload Firewall Configuration

```bash id="h0mdy8"
firewall-cmd --reload
```

---

## Verify SELinux Status

```bash id="rbfwwv"
getenforce
```

```bash id="2s92eo"
sestatus
```

> **⚠️ Critical**
>
> Never disable **SELinux** in a production environment.
>
> If troubleshooting requires SELinux to be set to **Permissive**, restore it to **Enforcing** immediately after the issue has been resolved.

# ✅ Operational Best Practices

Follow these recommendations during every patching and hardening cycle.

---

## 🧪 Validate in a Non-Production Environment

Always test operating system patches in a:

* Staging environment
* User Acceptance Testing (UAT) environment

before deploying them to production.

---

## 📄 Maintain a Rollback Plan

Every change request must include a documented rollback procedure that allows systems to be restored if patching fails.

---

## ⚙️ Automate Patching

Use **Ansible** to automate patch deployment across large server environments.

Benefits include:

* Reduced manual effort
* Consistent deployments
* Lower risk of human error
* Faster maintenance windows

---

## 🔍 Perform Post-Patch Vulnerability Scans

After patch installation, rerun vulnerability assessments using approved security tools such as:

* Nessus
* Qualys

Confirm that identified vulnerabilities have been successfully remediated.

---

## 📊 Maintain a Patch Compliance Matrix

Track patch status for every managed server.

Recommended fields include:

| Server | Patch Date | Patch Level | Approved By |
| ------ | ---------- | ----------- | ----------- |

Maintaining this matrix supports audit readiness and compliance reporting.

---

## 🛡️ Follow CIS Benchmark Standards

Implement **CIS Benchmark Level 1** as the minimum security baseline for all production Linux servers unless client requirements specify stricter controls.

---

## 🔄 Schedule Kernel Reboots

Kernel updates typically require a system reboot.

Never apply kernel patches without:

* An approved maintenance window
* Coordination with the application team
* Notification to affected stakeholders
