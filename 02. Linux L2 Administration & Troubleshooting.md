## Overview

This lab provides hands-on procedures for performing **Linux Level 2 (L2) administration, system validation, and troubleshooting** on production Linux servers. It focuses on routine health checks, remote administration, service management, authentication auditing, and process management while following enterprise operational best practices.

---

# 🎯 Objective

Perform **L2-level administration, configuration, monitoring, and troubleshooting** on Linux production servers running supported distributions.

---

This guide applies to the following operating systems and environments:

| Operating System                | Supported Versions |

| Red Hat Enterprise Linux (RHEL) | 7 / 8 / 9          |
| CentOS                          | 7 / 8              |
| Ubuntu Server                   | 20.04 / 22.04      |

**Environment:** Production Servers & Customer Environments

# Task 2.1 – Server Health Check Procedure

The following checks should be performed at the beginning of every maintenance activity or when investigating a server-related incident.

---

# 1. Verify System Uptime and Server Load

Check the system uptime, logged-in users, and server load averages.

### Display System Uptime

```bash
uptime
```

### Display Logged-in Users

```bash
w
```

### Verify

* System uptime
* Number of active users
* Load Average:

  * 1 minute
  * 5 minutes
  * 15 minutes

> **⚠️ Warning**
>
> If the **load average exceeds the total number of CPU cores**, the server may be experiencing high resource utilization.
>
> Investigate immediately to identify CPU-intensive processes or resource bottlenecks.

---

# 2. Verify Operating System and Kernel Version

Confirm that the operating system information matches the organization's Configuration Management Database (CMDB).

### Display OS Information

```bash
cat /etc/os-release
```

### Display Kernel Version

```bash
uname -r
```

### Display Host Information

```bash
hostnamectl
```

### Verify

* Operating System version
* Kernel version
* Hostname
* CMDB inventory information

---

# 3. Verify Running Services

Review the status of all active system services.

### List Running Services

```bash
systemctl list-units --type=service --state=running
```

### Check Service Status

```bash
systemctl status
```

### Identify Failed Services

```bash
systemctl --failed
```

If a service has failed, verify the approved change record before restarting it.

### Restart a Service

```bash
systemctl restart <service_name>
```

### Review Service Logs

```bash
journalctl -u <service_name> -n 50 --no-pager
```

> **Note**
>
> Always review service logs before restarting a production service.

---

# 4. Perform User and Authentication Audit

Review currently logged-in users and recent login activity.

### Display Logged-in Users

```bash
who
```

### Review Recent Login History

```bash
last | head -20
```

### List Interactive User Accounts

```bash
cat /etc/passwd | grep -v nologin | grep -v false
```

### Check Failed Login Attempts (RHEL / CentOS)

```bash
grep 'Failed password' /var/log/secure | tail -20
```

### Check Failed Login Attempts (Ubuntu)

```bash
grep 'Failed password' /var/log/auth.log | tail -20
```

### Verify

* Unexpected user logins
* Unauthorized root access
* Repeated authentication failures
* Suspicious login activity

# Task 2.2 – Remote Access & Basic Administration

---

# 5. Verify Secure SSH Access

Use SSH key-based authentication whenever possible.

### Connect to a Remote Server

```bash
ssh -i ~/.ssh/id_rsa username@server_ip
```

### Verify SSH Service

Check that the SSH daemon is running and listening on the expected port.

```bash
ss -tlnp | grep sshd
```

### Review SSH Configuration

```bash
cat /etc/ssh/sshd_config | grep -E 'Port|PermitRoot|PasswordAuth'
```

### Verify

* SSH service status
* Listening port
* Root login configuration
* Password authentication settings

---

# 6. File and Directory Management

Review disk usage and identify large or outdated log files.

### List Log Directory Contents

```bash
ls -lath /var/log/
```

### Find Large Log Files

```bash
find / -name '*.log' -mtime +30 -size +100M 2>/dev/null
```

### Display Largest Log Directories

```bash
du -sh /var/log/* | sort -rh | head -10
```

> **⚠️ Important**
>
> Never delete application or system files without approval from the application owner or support team.
>
> Archive old files instead of deleting them.

### Archive Old Log Files

```bash
tar -czf /backup/old_logs_$(date +%F).tar.gz /var/log/app/
```

---

# 7. Process Management

Identify resource-intensive processes and manage process priorities when necessary.

### Display Top CPU-Consuming Processes

```bash
ps aux --sort=-%cpu | head -15
```

### View System Resource Usage

```bash
top -bn1 | head -20
```

### Terminate a Process

```bash
kill -9 <PID>
```

> **⚠️ Warning**
>
> Use `kill -9` **only after obtaining the required approval** and confirming that terminating the process will not impact production services.

### Start a Process with Lower Priority

```bash
nice -n 10 <command>
```

### Change Priority of a Running Process

```bash
renice -n 15 -p <PID>
```

# ✅ Operational Best Practices

Follow these best practices during all production support activities.

## 🔄 Production Change Management

* Never restart production services without:

  * A valid change ticket, or
  * Explicit approval from the responsible manager.

---

## 📸 Capture Evidence

Before making any configuration or service changes:

* Capture screenshots, **or**
* Save command output for documentation and audit purposes.

Evidence is essential for troubleshooting and change validation.

---

## 🔐 Use Privileged Access Responsibly

Avoid switching to a permanent root shell.

**Preferred**

```bash
sudo <command>
```

**Avoid**

```bash
sudo -i
```

Use elevated privileges only when required.

---

## 📦 Validate Patch Compliance

Before performing maintenance:

* Verify the installed operating system version.
* Confirm the kernel version.
* Compare both against the client's approved patch baseline.

---

## 📋 Review CMDB Information

Before rebooting or making service-impacting changes:

* Verify the server information in the CMDB.
* Confirm whether an active maintenance window exists.
* Review any operational restrictions.

---

## 🖥️ Protect Long-Running Sessions

Use a terminal session manager when performing lengthy administrative tasks.

Recommended tools:

* `screen`
* `tmux`

This prevents interruptions caused by network disconnections or terminal session loss.
