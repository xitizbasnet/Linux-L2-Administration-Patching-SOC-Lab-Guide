## Overview

This lab provides standard operating procedures for monitoring enterprise infrastructure using the **SolarWinds Orion Platform**. It covers dashboard operations, node health verification, alert triage, ITIL-based incident escalation, and custom alert configuration to ensure timely incident detection and resolution.

These procedures are intended for Linux L2 Engineers responsible for infrastructure monitoring and operational support.

---

# 🎯 Objective

Perform monitoring and incident management activities by:

* Monitoring infrastructure through SolarWinds Orion
* Reviewing and responding to alerts
* Performing node health verification
* Following ITIL-based incident escalation procedures
* Configuring custom monitoring alerts
* Maintaining operational visibility and SLA compliance

---

# Task 8.1 – SolarWinds Dashboard Operations

Review monitoring dashboards and verify infrastructure health before responding to alerts.

---

# 1. Login & Dashboard Review

Access the **SolarWinds Orion Web Console**.

```text
https://solarwinds.company.com
```

After logging in:

* Navigate to **My Dashboard**.
* Review the following information:

  * Node Status
  * Alert Summary
  * Event Log
* Filter alerts using:

  * **Severity:** Critical / Warning
  * **Assignment Group:** Linux L2
  * **Status:** Active

> **⚠️ Important**
>
> Do **not** acknowledge an alert until you have reviewed and fully understood the condition it reports.

---

# 2. Verify Node Health

Navigate through the SolarWinds console:

```text
Manage → Nodes → Filter by "Down" or "Warning"
```

For each affected node, review:

* CPU Utilization
* Memory Utilization
* Disk Usage
* Packet Loss
* Response Time

Next:

* Open the **Events** tab.
* Review recent status changes and event history.
* Cross-validate the alert by connecting to the affected server via SSH to confirm its actual operational state.

---

# Task 8.2 – Alert Triage & Response

Classify alerts according to severity and respond within the required SLA.

---

# 3. Classify Alerts

Use the following severity guidelines when reviewing SolarWinds alerts.

| Severity           | Typical Conditions                             | Required Action                                       |
| ------------------ | ---------------------------------------------- | ----------------------------------------------------- |
| 🔴 **Critical**    | CPU > 95%, Disk > 95%, Node Down               | Immediate Priority 1 (P1) or Priority 2 (P2) response |
| 🟡 **Warning**     | CPU > 80%, Disk > 85%, High Memory Utilization | Investigate within **30 minutes**                     |
| 🔵 **Information** | Service Restart, Configuration Change          | Record the event and continue monitoring              |

After reviewing the alert:

1. Acknowledge the alert in SolarWinds.
2. Create a **ServiceNow Incident**.
3. Link the ServiceNow ticket to the corresponding SolarWinds Alert Reference ID.

---

# 4. Follow the ITIL-Based Escalation Process

Escalate incidents according to the organization's ITIL incident management workflow.

| Support Level  | Responsibilities                                                                  | Expected Time     |
| -------------- | --------------------------------------------------------------------------------- | ----------------- |
| **L1 Support** | Acknowledge the alert, perform initial checks, and create the incident ticket.    | **0–15 minutes**  |
| **L2 Support** | Diagnose the issue, attempt resolution, and update the incident every 30 minutes. | **15–60 minutes** |
| **L3 Support** | Escalate when the issue exceeds L2 capabilities or is approaching an SLA breach.  | **60+ minutes**   |

---

## Approved Escalation Methods

Escalate incidents using one or more of the following communication methods:

* 📞 Phone Call
* 💬 Microsoft Teams / Slack
* 🎫 ServiceNow Escalation Function

---

## Include the Following Information During Escalation

Every escalation should include:

* Server Name
* IP Address
* Alert Details
* Troubleshooting Steps Performed
* Time of Escalation

> **⚠️ Important**
>
> Never bypass an escalation level without documenting the reason.
>
> Maintaining a complete escalation audit trail is mandatory.

---

# 5. Configure a Custom Alert (L2 Administration)

Create custom alerts to improve proactive monitoring.

Navigate to:

```text
Alerts & Activity → Manage Alerts → Create New Alert
```

Configure the following alert condition:

| Setting       | Value               |
| ------------- | ------------------- |
| **Object**    | Node                |
| **Condition** | CPU Load > 90%      |
| **Duration**  | More than 5 minutes |

Configure notification actions:

* Send an email to:

```text
ops-team@company.com
```

* Automatically create a ServiceNow incident.

---

## Validate the Alert

Before enabling the alert in production:

* Test the alert in a **non-production environment**.
* Simulate the alert condition.
* Verify that:

  * Email notifications are delivered.
  * ServiceNow ticket creation is successful.
  * Alert thresholds trigger correctly.

# ✅ Operational Best Practices

Follow these recommendations to improve monitoring efficiency and reduce false positives.

---

## 📌 Acknowledge Every Alert

Acknowledge **every alert**, even if it automatically clears.

This confirms that:

* The alert was reviewed.
* Appropriate validation was performed.
* Operational accountability is maintained.

---

## 🛠️ Configure Alert Suppression

Create alert suppression windows during:

* Planned maintenance
* Server patching
* Infrastructure upgrades
* Scheduled outages

This helps reduce unnecessary alerts and prevents alert fatigue.

---

## 🌐 Use Network Atlas

Use **SolarWinds Network Atlas** to visualize:

* Network topology
* Device relationships
* Connectivity paths

Topology views assist in identifying upstream failures and accelerating root cause analysis.

---

## 📊 Customize Your Dashboard

Configure your dashboard to display only the infrastructure that you manage.

Benefits include:

* Reduced alert noise
* Faster incident identification
* Improved operational focus
* Reduced alert fatigue

---

## 🔍 Correlate Alerts Before Creating Incidents

Before raising a production incident:

* Review SolarWinds alerts.
* Verify operating system logs.
* Review application logs.
* Confirm that the issue is genuine.

Correlating multiple data sources helps eliminate false-positive Priority 1 (P1) incidents.

---

## 📝 Record the Alert ID

Always include the **SolarWinds Alert ID** in the corresponding **ServiceNow Incident**.

Maintaining this reference improves:

* Traceability
* Incident correlation
* Audit readiness
* Historical reporting
