## Overview

This lab defines standardized procedures for preparing **daily shift reports**, **structured handover notes**, and **Root Cause Analysis (RCA) documentation** for P1/P2 incidents.

It ensures operational consistency, audit readiness, and continuous service improvement in a 24×7 IT operations environment.

---

# 🎯 Objective

Prepare and maintain high-quality operational documentation including:

* Daily shift reports for operational tracking
* Structured shift handover notes
* RCA documentation for critical incidents
* 5-Why analysis for root cause identification
* Known Error Database (KEDB) entries
* Stakeholder communication records

# Task 12.1 – Daily Shift Report Preparation

---

# 1. Shift Report Structure

Each shift report must include the following standardized sections:

---

## 1. Shift Summary

Include:

* Date
* Shift type (Day / Evening / Night)
* Engineer name
* Team name

---

## 2. Incidents Handled

Document all incidents:

* INC number
* Server / CI name
* Issue description
* Priority
* Status
* Resolution time

---

## 3. Changes Executed

Include all changes performed:

* CRQ number
* Server affected
* Change type
* Outcome (Success / Failed)

---

## 4. Backup Status

Track backup activities:

* Backup job name
* Server
* Status (Success / Failure)
* Backup size

---

## 5. Monitoring Alerts

Summarize alerts:

* Count by severity (Critical / Warning / Info)
* Notable alerts
* Actions taken

---

## 6. Pending Items

List incomplete tasks:

* Task description
* Reason for delay
* Assigned next shift owner

---

## 7. Remarks

Include:

* Unusual system behavior
* Early warning indicators
* Risk observations

---

# Task 12.2 – Root Cause Analysis (RCA) Document

---

# 3. RCA Structure for P1/P2 Incidents

Each RCA document must include:

---

## 1. Incident Summary

* Incident ID (INC#)
* Server / CI
* Impact description
* Duration of outage

---

## 2. Timeline of Events

* Chronological sequence of events
* Include exact timestamps

---

## 3. Root Cause

* Clear technical explanation of the failure
* Direct cause of service impact

---

## 4. 5-Why Analysis

Systematic breakdown of root cause using iterative questioning

---

## 5. Immediate Fix

* Actions taken to restore service
* Temporary mitigation steps

---

## 6. Permanent Fix

* Long-term corrective actions
* Prevent recurrence of the issue

---

## 7. Action Items

* Assigned owners
* Tasks to be completed
* Due dates (SMART format)

---

## 8. Lessons Learned

* Improvements for future incidents
* Process gaps identified
* Preventive recommendations

---

# 4. 5-Why Analysis Example – Disk Full P1 Incident

---

## Problem Statement

Disk space exhaustion on `/var` filesystem leading to service outage.

---

## 5-Why Breakdown

* **Why 1:** `/var` filesystem reached 100% utilization
* **Why 2:** Application logs were not being rotated
* **Why 3:** `logrotate` configuration was missing application log path
* **Why 4:** Post-deployment checklist did not include logrotate validation
* **Why 5:** No standardized post-deployment checklist existed for this application

---

## Root Cause

Absence of a standardized post-deployment checklist.

---

## Permanent Fix

* Implement a standardized post-deployment checklist
* Include mandatory logrotate configuration validation
* Enforce checklist via deployment governance

---

# 5. KEDB – Known Error Database Entry

After RCA approval, create a Known Error record in ServiceNow:

```text id="kedb1"
Problem → Known Error → Create
```

---

## Example KEDB Entry

| Field             | Value                                               |
| ----------------- | --------------------------------------------------- |
| Known Error Title | [APP] Disk Full due to missing logrotate config     |
| Workaround        | Manually truncate logs or temporarily extend LV     |
| Permanent Fix     | Deploy standard logrotate configuration via Ansible |
| Related Incidents | INC001234, INC001567                                |

---

## Usage

* Link KEDB entries to recurring incidents
* Improve resolution speed
* Reduce repetitive troubleshooting effort

# ✅ Operational Best Practices

---

## ⏱ RCA Timelines

* P1 RCA must be submitted within **24 hours**
* P2 RCA must be submitted within **48 hours**

---

## 🤝 Collaborative RCA Process

* Perform 5-Why analysis as a team exercise
* Involve all resolver groups for accuracy

---

## 📄 Mandatory Reporting

* Shift reports must never be skipped
* Even quiet shifts require full documentation

---

## 🧠 Value of RCA

* RCA is a critical operational deliverable
* Prevents recurrence of major incidents
* Drives continuous service improvement

---

## 📊 Standardization Requirement

* Always use templates for:

  * Shift reports
  * RCA documents
* Ensures consistency and audit readiness

---

## 📌 ServiceNow Integration

* Track RCA action items as tasks
* Assign owners and deadlines
* Ensure accountability and closure tracking

---

## 📚 Knowledge Sharing

* Share RCA findings in weekly “Lessons Learned” sessions
* Improve team-wide operational maturity
