## Overview

This lab provides standard troubleshooting procedures for diagnosing and resolving common Linux operating system issues in production environments. It covers operating system logs, service failures, Logical Volume Manager (LVM), Multipath I/O, file system integrity, and disk space management.

The procedures are intended for Linux L2 Engineers responsible for maintaining system availability and resolving infrastructure-related incidents.

---

# 🎯 Objective

Perform Level 2 troubleshooting activities to:

* Diagnose operating system issues
* Investigate service failures
* Administer and extend LVM storage
* Verify Multipath I/O health
* Repair file system issues
* Resolve disk space emergencies
* Maintain production system stability

# Task 6.1 – Operating System & Service Troubleshooting

Perform the following checks when investigating operating system instability or service failures.

---

# 1. Analyze System Logs

System logs provide valuable information about hardware failures, kernel events, and service errors.

## Review Kernel Messages

```bash
dmesg -T | tail -50
```

Displays the latest kernel ring buffer messages with timestamps.

---

## Review Recent System Journal Entries

```bash
journalctl -xe
```

Shows recent system events and errors.

---

## Review Logs from the Last Hour

```bash
journalctl --since "1 hour ago"
```

Useful for correlating events with recently reported incidents.

---

## Monitor System Logs (RHEL / CentOS)

```bash
tail -f /var/log/messages
```

---

## Monitor System Logs (Ubuntu)

```bash
tail -f /var/log/syslog
```

---

### Verify

Look for the following critical events:

* Out of Memory (OOM) Killer
* Hardware failures
* Kernel segmentation faults (Segmentation Faults)
* Disk I/O errors
* Storage-related warnings

> **⚠️ Important**
>
> Kernel messages often provide the first indication of hardware or storage failures.

---

# 2. Diagnose Service Failures

When a service fails to start or becomes unavailable, perform the following checks.

## Check Service Status

```bash
systemctl status --full <service_name>
```

---

## Review Service Logs

```bash
journalctl -u <service_name> --since "30 min ago"
```

---

## Display Service Dependencies

```bash
systemctl list-dependencies <service_name>
```

Verify that all required dependent services are running.

---

## Check for Port Conflicts

If a service cannot bind to its configured port:

```bash
ss -tlnp | grep <port>
```

or

```bash
fuser <port>/tcp
```

Identify the process currently using the required port.

# Task 6.2 – LVM Administration

Manage and expand Logical Volume Manager (LVM) storage when additional capacity is required.

---

# 3. Review LVM Configuration

Display the current storage layout.

## Physical Volumes

```bash
pvs
```

---

## Volume Groups

```bash
vgs
```

---

## Logical Volumes

```bash
lvs
```

---

## Block Device Layout

```bash
lsblk
```

Review the relationship between disks, partitions, and logical volumes.

---

# 4. Extend an LVM Volume

A common L2 administrative task is extending an existing logical volume.

---

## Step 1 – Verify Available Free Space

Check whether the Volume Group contains available Physical Extents (PE).

```bash
vgdisplay | grep "Free PE"
```

---

## Step 2 – Extend the Logical Volume

Increase the logical volume by a fixed size.

```bash
lvextend -L +10G /dev/vg_data/lv_app
```

Or allocate all remaining free space.

```bash
lvextend -l +100%FREE /dev/vg_data/lv_app
```

---

## Step 3 – Resize the File System

No unmount is required for online resizing of supported file systems.

### ext4

```bash
resize2fs /dev/vg_data/lv_app
```

### XFS

```bash
xfs_growfs /data
```

> **Note**
>
> For XFS, use the **mount point**, not the device name.

---

## Step 4 – Verify the New Capacity

```bash
df -h /data
```

Confirm that the filesystem reflects the expanded storage.

# Task 6.3 – Multipath Troubleshooting

Verify storage path redundancy and diagnose Multipath-related issues.

---

# 5. Verify Multipath Status

Display all configured storage paths.

```bash
multipath -ll
```

---

## Display Individual Paths

```bash
multipathd show paths
```

---

## Review Multipath Configuration

```bash
cat /etc/multipath.conf
```

---

### Verify

Review for:

* Active paths
* Failed paths
* Path priorities
* Device mappings

> **⚠️ Important**
>
> Faulty storage paths commonly generate Disk I/O errors in `dmesg`.
>
> Report failed paths to the Storage or SAN Team for further investigation.

---

## Reload Multipath Configuration

After modifying the Multipath configuration:

Restart the Multipath daemon.

```bash
systemctl restart multipathd
```

Reload Multipath device maps.

```bash
multipath -r
```

# Task 6.4 – File System Checks & Repair

Perform file system integrity checks and recover from filesystem-related issues.

---

# 6. Perform a File System Check (fsck)

> **⚠️ Critical**
>
> Never execute `fsck` on a mounted filesystem.
>
> Unmount the filesystem or boot into rescue mode before running file system repair.

---

## Unmount the Filesystem

```bash
umount /dev/sdb1
```

---

## Repair ext4 File System

```bash
fsck -y /dev/sdb1
```

Automatically repairs detected filesystem errors.

---

## Repair XFS File System

```bash
xfs_repair /dev/sdb1
```

---

## Verify Mount Configuration

Current mounted filesystems:

```bash
cat /proc/mounts
```

Persistent mount configuration:

```bash
cat /etc/fstab
```

Review mount options and ensure all required filesystems are configured correctly.

---

# 7. Resolve Disk Full Emergencies

When a filesystem becomes full, identify the source of disk usage before taking corrective action.

---

## Identify Large Directories

```bash
du -sh /* 2>/dev/null | sort -rh | head -10
```

---

## Locate Large Files

```bash
find / -xdev -type f -size +500M 2>/dev/null
```

---

## Safely Clear Large Log Files

Instead of deleting an active log file, truncate it.

```bash
> /var/log/application.log
```

---

## Reduce System Journal Size

```bash
journalctl --vacuum-size=500M
```

Removes older journal entries while preserving recent logs.

---

## Identify Deleted Files Still Consuming Space

```bash
lsof +L1
```

> **Important**
>
> Deleting an open file does **not** immediately release disk space.
>
> Disk space is reclaimed only after the owning process closes the file.

# ✅ Operational Best Practices

Follow these recommendations during Linux troubleshooting and storage administration.

---

## 🚫 Avoid Direct File Deletion

Never execute destructive commands such as:

```bash
rm -rf
```

on production systems.

Instead:

1. Move files to a temporary location.
2. Verify they are no longer required.
3. Delete them only after confirmation.

---

## 📝 Document All LVM Changes

Every storage modification should include:

* Physical Volume (PV)
* Volume Group (VG)
* Logical Volume (LV)
* Previous size
* New size
* Change Request reference

Record all changes in the approved change ticket.

---

## 💾 Engage the Storage Team for Multipath Issues

Do not attempt to repair storage path failures without involving the Storage or SAN Team.

Multipath issues frequently originate from:

* SAN connectivity
* Fibre Channel switches
* Storage arrays
* HBA configuration

---

## ⚙️ Configure Resilient Mount Options

For non-critical filesystems, configure the `nofail` option in `/etc/fstab`.

This allows the operating system to continue booting even if the filesystem is unavailable.

---

## 🔄 Configure Log Rotation

Ensure all application logs are managed using **logrotate**.

Proper log rotation helps prevent:

* `/var/log` filesystem exhaustion
* Unexpected application failures
* Priority 1 (P1) incidents caused by disk-full conditions

---

## 📚 Maintain an Operational Runbook

Create and maintain a runbook documenting the **Top 10 Linux issues** commonly encountered within your environment.

The runbook should include:

* Symptoms
* Root cause
* Resolution steps
* Escalation procedures
* Reference commands

Keeping this documentation current improves troubleshooting efficiency and reduces incident resolution time.
