## Overview

This lab defines standardized ITIL-based operational procedures for managing **incidents, problems, and changes**, along with structured **shift handover practices** in a 24×7 enterprise support environment.

It aligns with **ITIL v4 best practices** and ServiceNow ITSM workflows to ensure service reliability, accountability, and continuity between shifts.

---

# 🎯 Objective

Apply ITIL-based processes to:

* Manage incident lifecycle (identification → resolution → closure)
* Perform structured problem management and RCA linkage
* Execute change management via CAB-approved workflows
* Conduct effective shift handovers in 24×7 operations

# Task 11.1 – Incident Management Lifecycle

---

# 1. Incident Identification & Logging

Incidents may originate from multiple sources:

* 📡 SolarWinds alerts
* 📞 User-reported issues
* 📧 Monitoring system notifications
* 👨‍💻 Shift engineer observations

---

## Logging Requirement

All incidents must be logged in **ServiceNow within 5 minutes** of detection.

---

## Mandatory Fields

Ensure the following fields are correctly populated:

* Priority
* Category
* Assignment Group
* Affected Configuration Item (CI) from CMDB

> ⚠️ Accurate CI mapping is critical for impact analysis and reporting.

---

# 2. Incident Investigation & Diagnosis

---

## Root Cause Analysis Approach

Apply the **5-Why technique** to systematically identify root causes.

---

## Investigation Steps

Check the following:

* Recent changes (last 24–48 hours)
* Patch and update history
* Monitoring baseline deviations
* System logs and performance trends

---

## Communication & Updates

* Update **Work Notes every 30 minutes (P1)**
* Update **Work Notes every 1 hour (P2)**

Include:

* Progress updates
* Findings
* Commands executed
* Observations

---

## Collaboration Requirement

* Engage resolver groups early
* Do not wait for SLA breach before escalation

---

# 3. Incident Resolution & Closure

---

## Resolution Process

* Apply corrective fix
* Validate system recovery via monitoring tools
* Confirm with end user or stakeholder

---

## Mandatory Closure Details

Update the incident with:

* Resolution Notes
* Root Cause
* Actions Taken
* CI Updates (if applicable)

---

## Closure Rule

> 🚨 An incident must only be closed after **user/customer confirmation** of resolution.

---

# Task 11.2 – Problem Management

---

# 4. Converting Incidents to Problem Records

Recurring issues must be formally tracked.

---

## Problem Identification Criteria

Create a Problem record if:

* Same issue occurs **3 or more times within 30 days**

---

## Problem Creation

Navigate in ServiceNow:

```text id="pb1"
Problem → Create New → Link Related Incidents
```

---

## Assignment & Resolution Strategy

* Assign to specialist teams
* Identify:

  * Workaround
  * Permanent fix

---

## Known Error Database (KEDB)

* Document workaround in KEDB
* Ensure visibility for future incidents

---

# Task 11.3 – Change Management

---

# 5. Change Request (CRQ) Lifecycle

All infrastructure changes must follow approved change processes.

---

## Types of Changes

| Type                 | Description                                                   |
| -------------------- | ------------------------------------------------------------- |
| **Standard Change**  | Pre-approved, low risk (e.g., routine patching)               |
| **Normal Change**    | Requires CAB approval (e.g., server configuration changes)    |
| **Emergency Change** | P1 resolution; documented post-implementation within 24 hours |

---

## CRQ Mandatory Requirements

Each Change Request must include:

* Implementation Plan
* Rollback Plan
* Test Plan
* Risk Assessment

---

## Change Communication

* Notify stakeholders **48–72 hours in advance**
* Schedule within approved maintenance window

---

## Post-Change Updates

After implementation, update CRQ with:

* Actual Start Time
* Actual End Time
* Outcome (Success / Failed)

---

# Task 11.4 – Shift Handover Procedure

---

# 6. Effective Shift Handover

Begin preparation **30 minutes before shift end**.

---

## Mandatory Handover Content

Include the following:

* 🔴 Active P1/P2 incidents with current status and latest actions
* 📌 Pending tasks with deadlines
* 🛠️ Ongoing changes or maintenance activities
* 📞 Escalations already initiated
* 👀 Known issues to monitor in the next 2 hours

---

## Handover Communication

* Send handover email to:

  * Incoming engineer
  * Team lead
  * Group mailbox

---

## Verbal Handover Requirement

* Conduct a **Teams/voice handover** for all active P1 incidents
* Email-only handover is not sufficient for critical incidents

---

# ✅ Operational Best Practices

---

## 🚨 P1 Incident Handling

* Establish bridge call within **15 minutes**
* Avoid isolated troubleshooting on critical incidents

---

## ⏱ Shift Handover Quality

* Effective handover duration: **10–15 minutes**
* Rushed handovers increase risk of missed incidents

---

## 📄 Standard Templates

* Always use a **standardized handover template**
* Ensures consistency and prevents data loss

---

## 📊 Root Cause Analysis Requirement

* Every resolved P1 incident must have RCA completed within **24–48 hours**

---

## 📝 Shift Logging Discipline

Maintain a shift log including:

* Timestamp
* Server / CI
* Issue description
* Actions taken
* Resolution status

---

## ⛔ Change Freeze Awareness

Be aware of organizational blackout periods:

* Month-end closures
* Year-end processing
* Business-critical windows
