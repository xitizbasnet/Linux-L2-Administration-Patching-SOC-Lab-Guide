## Overview

This lab provides standard procedures for monitoring and analyzing Linux server performance. It focuses on identifying performance bottlenecks related to **CPU, memory, disk I/O, system load, and network activity** using native Linux monitoring tools. These checks enable proactive issue detection, capacity planning, and timely incident response.

---

# 🎯 Objective

Monitor and analyze Linux system performance metrics to:

* Identify resource bottlenecks
* Detect abnormal system behavior
* Troubleshoot performance-related incidents
* Maintain optimal server health
* Support proactive infrastructure monitoring

---

# 📌 Scope

This lab applies to performance monitoring on Linux servers using standard system utilities.

| Category                 | Tools / Utilities                             |
| ------------------------ | --------------------------------------------- |
| **CPU Monitoring**       | `top`, `htop`, `mpstat`, `sar`, `pidstat`     |
| **Memory Monitoring**    | `free`, `vmstat`, `smem`, `/proc/meminfo`     |
| **Disk Monitoring**      | `df`, `du`, `iostat`, `iotop`, `lsblk`        |
| **Network Monitoring**   | `sar`, `ss`, `netstat`                        |
| **Performance Analysis** | System Load, I/O Statistics, Alert Thresholds |

# Task 5.1 – CPU Performance Analysis

Analyze processor utilization and identify processes consuming excessive CPU resources.

---

# 1. Monitor CPU Usage in Real Time

Use the following commands to observe CPU utilization.

## Display Interactive CPU Usage

```bash id="ywxv7m"
top -d 2
```

Refreshes every **2 seconds**.

---

## Interactive Process Viewer

```bash id="cl44ew"
htop
```

> **Note**
>
> Install `htop` if it is not already available on the server.

---

## Display Per-CPU Statistics

```bash id="myl13v"
mpstat -P ALL 2 5
```

Collects CPU statistics for every processor every **2 seconds** over **5 intervals**.

---

## Display CPU Utilization History

```bash id="0v59md"
sar -u 1 10
```

Captures **10 CPU utilization snapshots** at **1-second intervals**.

---

### Performance Indicators

| Metric                    | Interpretation                                                     |
| ------------------------- | ------------------------------------------------------------------ |
| **%iowait > 20%**         | Indicates potential disk I/O bottlenecks.                          |
| **High %sy (System CPU)** | May indicate excessive kernel activity or system-level processing. |

Investigate abnormal values promptly to determine the root cause.

---

# 2. Identify CPU-Intensive Processes

Locate processes consuming excessive CPU resources.

## Display Top CPU Consumers

```bash id="b67w8q"
ps aux --sort=-%cpu | head -10
```

---

## Monitor CPU Usage Per Process

```bash id="gj0kv9"
pidstat -u 2 5
```

> Requires the **sysstat** package.

---

## Display Top Processes

```bash id="rl3agp"
top -bn1 | grep -v '^top' | head -20
```

Review high CPU utilization before taking corrective action.

# Task 5.2 – Memory Analysis

Monitor physical memory utilization and identify memory-related issues.

---

# 3. Verify Memory Usage

## Display Memory Summary

```bash id="r65qpi"
free -h
```

---

## Display Memory Statistics

```bash id="o8lc8i"
vmstat -s
```

---

## Display Detailed Memory Information

```bash id="bm6jfc"
cat /proc/meminfo
```

### Memory Threshold

> **⚠️ Critical**
>
> If **available memory falls below 10%** of total physical memory, escalate the issue immediately and investigate possible memory leaks.

---

## Verify Swap Usage

High swap usage may indicate insufficient physical memory.

```bash id="uj6cw5"
swapon --show
```

---

## Monitor Swap Activity

```bash id="a6y0ef"
vmstat 1 10
```

Review the following columns:

* **si** (Swap In)
* **so** (Swap Out)

Frequent swap activity typically indicates memory pressure.

---

# 4. Identify Memory-Intensive Processes

Locate processes consuming excessive memory.

## Display Top Memory Consumers

```bash id="aax6ka"
ps aux --sort=-%mem | head -10
```

---

## Monitor Process Memory Usage

```bash id="9iqcd0"
pidstat -r 2 5
```

---

## Display Actual Memory Usage (PSS)

```bash id="umff2x"
smem -t -k -c pss -s pss | tail -10
```

Provides proportional memory usage (PSS), offering a more accurate view of per-process memory consumption.

# Task 5.3 – Disk & I/O Performance

Monitor storage utilization and identify disk-related performance issues.

---

# 5. Verify Disk Usage

## Display Mounted File Systems

```bash id="4eq2d4"
df -hT
```

---

## Identify Large Directories

```bash id="gvfe8d"
du -sh /var/* | sort -rh | head -10
```

---

## Display Block Devices

```bash id="1qutqx"
lsblk
```

### Disk Utilization Thresholds

| Disk Usage | Required Action                                           |
| ---------- | --------------------------------------------------------- |
| **80%**    | ⚠️ Generate an alert and begin investigation.             |
| **90%**    | 🚨 Escalate immediately according to incident procedures. |

> **Note**
>
> Rapid growth of **`/tmp`**, **`/`**, or **`/var`** is a common cause of **Priority 1 (P1)** production incidents.

---

# 6. Analyze Disk I/O Performance

Monitor disk throughput and latency.

## Display Extended I/O Statistics

```bash id="ecmztg"
iostat -xz 2 5
```

---

## Display Per-Process Disk Activity

```bash id="yv3kcq"
iotop -o
```

> Requires root privileges.

---

## Display Combined Performance Statistics

```bash id="kjlwmv"
dstat -cdngy 2
```

Displays combined statistics for:

* CPU
* Disk
* Network
* Memory

---

### Key Performance Metrics

| Metric    | Threshold          | Interpretation       |
| --------- | ------------------ | -------------------- |
| **%util** | Greater than 90%   | Disk saturation      |
| **await** | Greater than 20 ms | High storage latency |

Investigate storage devices, SAN, or network-attached storage when these thresholds are exceeded.

# ✅ Operational Best Practices

Follow these recommendations during routine performance monitoring and troubleshooting.

---

## 🚨 Configure Automated Alerts

Implement monitoring thresholds for proactive incident detection.

Recommended alert thresholds:

| Metric           | Threshold                       |
| ---------------- | ------------------------------- |
| CPU Utilization  | Above **90%** for **5 minutes** |
| Available Memory | Below **10%**                   |
| Disk Usage       | Above **85%**                   |

Configure automated ticket creation when thresholds are exceeded.

---

## 📈 Compare Historical Performance

Use historical performance data stored under:

```text
/var/log/sa/
```

Analyze historical trends to determine whether current resource usage deviates from the established baseline.

---

## 💽 Investigate High I/O Wait

If the following conditions occur simultaneously:

* High **%iowait**
* Low CPU utilization

The issue is likely related to:

* Disk subsystem performance
* SAN storage
* NFS storage
* Storage controller latency

Investigate storage infrastructure before optimizing CPU resources.

---

## 📊 Maintain Performance Baselines

Document baseline performance metrics for every production server, including:

* CPU utilization
* Memory usage
* Disk utilization
* System load
* Network throughput

Significant deviations from baseline values should trigger further investigation.

---

## ☕ Monitor Java-Based Applications

For Java applications:

* High memory utilization
* Frequent **OutOfMemoryError (OOM)** messages

may indicate an incorrectly configured JVM heap.

Review:

* Heap size
* Garbage Collection (GC)
* JVM startup parameters

---

## ⚠️ Review Kernel Messages

Always examine kernel logs alongside performance metrics.

```bash
dmesg
```

Kernel messages often reveal:

* Hardware failures
* Disk errors
* Memory issues
* Driver problems
* Filesystem warnings

These messages can significantly reduce troubleshooting time.
